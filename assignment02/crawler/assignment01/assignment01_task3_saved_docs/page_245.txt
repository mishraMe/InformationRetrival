<p>In <a href="/wiki/Computing" title="Computing">computing</a>, a <b>plug and play</b> device or <a href="/wiki/Computer_bus" class="mw-redirect" title="Computer bus">computer bus</a>, is one with a specification that facilitates the discovery of a hardware component in a system without the need for physical device configuration or user intervention in resolving resource conflicts.<sup id="cite_ref-pcguidedefinition_1-0" class="reference"><a href="#cite_note-pcguidedefinition-1">[1]</a></sup><sup id="cite_ref-pcmagdefinition_2-0" class="reference"><a href="#cite_note-pcmagdefinition-2">[2]</a></sup></p>
<p>Plug and play devices can be due to boot-time assignment of device resources and to <a href="/wiki/Hot_swapping" title="Hot swapping">hotplug</a> systems such as <a href="/wiki/USB" title="USB">USB</a> and <a href="/wiki/IEEE_1394" title="IEEE 1394">IEEE 1394</a> (FireWire).<sup id="cite_ref-techtermsdefinition_3-0" class="reference"><a href="#cite_note-techtermsdefinition-3">[3]</a></sup></p>

<h2><span class="mw-headline" id="History_of_device_configuration">History of device configuration</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=1" title="Edit section: History of device configuration">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a href="/wiki/File:IBM402plugboard.Shrigley.wireside.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/b7/IBM402plugboard.Shrigley.wireside.jpg/220px-IBM402plugboard.Shrigley.wireside.jpg" width="220" height="165" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/b7/IBM402plugboard.Shrigley.wireside.jpg/330px-IBM402plugboard.Shrigley.wireside.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/b7/IBM402plugboard.Shrigley.wireside.jpg/440px-IBM402plugboard.Shrigley.wireside.jpg 2x" data-file-width="1024" data-file-height="768" /></a>
<div class="thumbcaption">
<div class="magnify"><a href="/wiki/File:IBM402plugboard.Shrigley.wireside.jpg" class="internal" title="Enlarge"></a></div>
<a href="/wiki/IBM_402" title="IBM 402">IBM 402</a> Accounting Machine control panel wiring. This control panel was labeled "profit &amp; loss summary".</div>
</div>

<p>In the beginnings of data processing technology, the hardware was just a collection of modules, and the functions of those modules had to be linked to accommodate different calculating operations. This linking was usually done by connecting some wires between modules and disconnecting others. For many <a href="/wiki/Unit_record_equipment" title="Unit record equipment">mechanical data processing machines</a>, such as the IBM <a href="/wiki/Punched_card" title="Punched card">punched card</a> <a href="/wiki/Tabulating_machine" title="Tabulating machine">accounting machines</a>, their calculating operations were directed by the use of a quick-swap <a href="/wiki/Plugboard" title="Plugboard">control panel</a> wired to route signals between module sockets.</p>
<p>As general purpose computing devices developed, these connections and disconnections were instead used to specify locations in the system <a href="/wiki/Address_space" title="Address space">address space</a> where an expansion device should appear, in order for the device to be accessible by the <a href="/wiki/Central_processing_unit" title="Central processing unit">central processing unit</a>. If two or more of the same type of device were installed in one computer, it would be necessary to assign the second device to a separate, non-overlapping region of the system address space so that both could be accessible at the same time.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a href="/wiki/File:Microdigital_Super_Serial_Card_SSC.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/9a/Microdigital_Super_Serial_Card_SSC.jpg/220px-Microdigital_Super_Serial_Card_SSC.jpg" width="220" height="95" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/9a/Microdigital_Super_Serial_Card_SSC.jpg/330px-Microdigital_Super_Serial_Card_SSC.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/9a/Microdigital_Super_Serial_Card_SSC.jpg/440px-Microdigital_Super_Serial_Card_SSC.jpg 2x" data-file-width="2106" data-file-height="913" /></a>
<div class="thumbcaption">
<div class="magnify"><a href="/wiki/File:Microdigital_Super_Serial_Card_SSC.jpg" class="internal" title="Enlarge"></a></div>
Apple II serial interface card that required cutting and soldering to reconfigure. The user would cut the wire trace between the <b>&gt;&lt;</b> cones at X1 and X3 and solder the <b>&lt;&gt;</b> cones together at X2 and X4. Located vertically near center of board.</div>
</div>
</div>
<p>Some early microcomputing devices such as the <a href="/wiki/Apple_II" title="Apple II">Apple II</a> sometimes required the end-user to physically cut some wires and solder others together to make configuration changes. The changes were intended to be largely permanent for the life of the hardware.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2014)">citation needed</span></a></i>]</sup></p>
<p>As computers became more accessible to the general public, the need developed for more frequent changes to be made by computer users unskilled with using soldering irons. Rather than cutting and soldering connections, configuration was accomplished by <a href="/wiki/Jumper_(computing)" title="Jumper (computing)">jumpers</a> or <a href="/wiki/DIP_switch" title="DIP switch">DIP switches</a>.</p>
<p><br /></p>

<p>However, the process of configuring devices manually using jumpers or DIP switches, could be quite difficult, and there was usually no forgiveness for technical inexperience. Incorrect settings could render either the whole system or just the expansion device completely or partially inoperable. Some settings such as for <a href="/wiki/Interrupt_request" class="mw-redirect" title="Interrupt request">interrupts</a> (IRQ) or <a href="/wiki/Direct_memory_access" title="Direct memory access">direct memory access</a> (DMA) required knowledge of what configuration had been previously manually assigned to other devices, as well as what settings the main system itself may have been using. The system might still have seemed to work properly with an incorrect setting, until the IRQ or DMA was actually needed and the entire system suddenly froze and needed to be reset.</p>
<h2><span class="mw-headline" id="First_attempts_at_self-configuration">First attempts at self-configuration</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=2" title="Edit section: First attempts at self-configuration">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As computing devices spread further out into the general population, there was ever greater pressure developing to automate this configuration process, and relieve the end-user from the hassle and complexity of having to manually assign these many complex settings.</p>
<h2><span class="mw-headline" id="MSX">MSX</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=3" title="Edit section: MSX">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/MSX" title="MSX">MSX</a> system, released in 1983, was designed to be plug and play from ground up, and achieved this by a clever system of slots/subslots, where each had their own <a href="/wiki/Virtual_address_space" title="Virtual address space">virtual address space</a>, thus eliminating device addressing conflicts in its very source. No jumpers or any manual configuration was required, and the independent address space for each slot allowed very cheap &amp; commonplace chips to be used, alongside with cheap <a href="/wiki/Glue_logic" title="Glue logic">glue logic</a>. On the software site, the drivers and extensions were supplied in the cards own ROM, thus requiring no disks or any kind of user intervention to configure the software. The ROM extensions <a href="/wiki/Hardware_abstraction" title="Hardware abstraction">abstracted any hardware differences</a> and offered standard APIs as specified by <a href="/wiki/ASCII_Corporation" title="ASCII Corporation">ASCII Corporation</a>.</p>
<h3><span class="mw-headline" id="NuBus">NuBus</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=4" title="Edit section: NuBus">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:102px;"><a href="/wiki/File:Radius_PrecisionColor_Series_24X_1993_1_front.JPG" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/a/a6/Radius_PrecisionColor_Series_24X_1993_1_front.JPG/100px-Radius_PrecisionColor_Series_24X_1993_1_front.JPG" width="100" height="75" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/a/a6/Radius_PrecisionColor_Series_24X_1993_1_front.JPG/150px-Radius_PrecisionColor_Series_24X_1993_1_front.JPG 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/a/a6/Radius_PrecisionColor_Series_24X_1993_1_front.JPG/200px-Radius_PrecisionColor_Series_24X_1993_1_front.JPG 2x" data-file-width="2048" data-file-height="1536" /></a>
<div class="thumbcaption">
<div class="magnify"><a href="/wiki/File:Radius_PrecisionColor_Series_24X_1993_1_front.JPG" class="internal" title="Enlarge"></a></div>
A NuBus expansion card without jumpers or DIP switches.</div>
</div>
</div>
<p>In 1984, the <a href="/wiki/NuBus" title="NuBus">NuBus</a> architecture was developed by the Massachusetts Institute of Technology (MIT) as a platform agnostic peripheral interface that fully automated device configuration. The specification was sufficiently intelligent that it could work with both <a href="/wiki/Big_endian" class="mw-redirect" title="Big endian">big endian</a> and <a href="/wiki/Little_endian" class="mw-redirect" title="Little endian">little endian</a> computer platforms that had previously been mutually incompatible. However, this agnostic approach increased interfacing complexity and required support chips on every device which in the 1980s was expensive to do, so the technology did not gain widespread support.</p>
<h3><span class="mw-headline" id="Zorro_bus">Zorro bus</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=5" title="Edit section: Zorro bus">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>In 1984, Commodore developed the AutoConfig protocol and the Zorro expansion bus for its <a href="/wiki/Amiga" title="Amiga">Amiga</a> line of expandable computers. The first public appearance was in the CES computer show at Las Vegas in 1985, with the so-called "Lorraine" prototype. Like NuBus, Zorro devices had absolutely no jumpers or DIP switches. The Zorro architecture did not spread to general computing use outside of the Amiga product line, but was eventually upgraded for the later iteration of Amiga computers.</p>
<h3><span class="mw-headline" id="Micro-Channel_Architecture">Micro-Channel Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=6" title="Edit section: Micro-Channel Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright">
<div class="thumbinner" style="width:202px;"><a href="/wiki/File:MCA_NIC_IBM_83X9648.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/55/MCA_NIC_IBM_83X9648.jpg/200px-MCA_NIC_IBM_83X9648.jpg" width="200" height="79" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/55/MCA_NIC_IBM_83X9648.jpg/300px-MCA_NIC_IBM_83X9648.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/55/MCA_NIC_IBM_83X9648.jpg/400px-MCA_NIC_IBM_83X9648.jpg 2x" data-file-width="2526" data-file-height="1002" /></a>
<div class="thumbcaption">
<div class="magnify"><a href="/wiki/File:MCA_NIC_IBM_83X9648.jpg" class="internal" title="Enlarge"></a></div>
An MCA expansion card without jumpers or DIP switches.</div>
</div>
</div>
<p>In 1987, IBM released an update to the <a href="/wiki/IBM_PC" class="mw-redirect" title="IBM PC">IBM PC</a> known as the <a href="/wiki/Personal_System/2" class="mw-redirect" title="Personal System/2">Personal System/2</a> line of computers using the <a href="/wiki/Micro_Channel_Architecture" class="mw-redirect" title="Micro Channel Architecture">Micro Channel Architecture</a>. The PS/2 was capable of totally automatic self-configuration. Every piece of expansion hardware was issued with a floppy disk containing a special file used to <a href="/wiki/Auto-configure" class="mw-redirect" title="Auto-configure">auto-configure</a> the hardware to work with the computer. The user would install the device, turn on the computer, load the configuration information from the disk, and the hardware automatically assigned interrupts, DMA, and so forth.</p>
<p>However, the disks posed a problem if they were damaged or lost, as the only options at the time to obtain replacements was via postal mail or IBM's dialup <a href="/wiki/Bulletin_board_system" title="Bulletin board system">BBS</a> service. Without the disks, any new hardware would be completely useless and the computer would not boot at all until the unconfigured device was removed.</p>
<p>Microchannel did not gain widespread support, because IBM wanted to exclude clone manufacturers from this next generation computing platform. Anyone developing for MCA had to sign non-disclosure agreements and pay royalties to IBM for each device sold, putting a price premium on MCA devices. End-users and the clone manufacturers revolted against IBM and developed their own open standards bus, known as EISA. Consequently, MCA receded in use except in IBM's mainframes.</p>
<h2><span class="mw-headline" id="ISA_and_PCI_self-configuration">ISA and PCI self-configuration</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=7" title="Edit section: ISA and PCI self-configuration">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In time, many <a href="/wiki/Industry_Standard_Architecture" title="Industry Standard Architecture">Industry Standard Architecture</a> (ISA) cards incorporated, through proprietary and varied techniques, hardware to self-configure or to provide for software configuration; often, the card came with a configuration program on disk that could automatically set the software-configurable (but not itself self-configuring) hardware. Some cards had both jumpers and software-configuration, with some settings controlled by each; this compromise reduced the number of jumpers that had to be set, while avoiding great expense for certain settings, e.g. nonvolatile registers for a base address setting. The problems of required jumpers continued on, but slowly diminished as more and more devices, both ISA and other types, included extra self-configuration hardware. However, these efforts still did not solve the problem of making sure the end-user has the appropriate software driver for the hardware.</p>
<p>ISA PnP or (legacy) Plug &amp; Play ISA was a plug-n-play system that used a combination of modifications to hardware, the system BIOS, and operating system software to automatically manage resource allocations. It was superseded by the <a href="/wiki/Conventional_PCI" title="Conventional PCI">PCI</a> bus during the mid-1990s.</p>
<h2><span class="mw-headline" id="Plug_and_Play">Plug and Play</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=8" title="Edit section: Plug and Play">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div role="note" class="hatnote">Main article: <a href="/wiki/Legacy_Plug_and_Play" title="Legacy Plug and Play">Legacy Plug and Play</a></div>
<p>In 1995, Microsoft released <a href="/wiki/Windows_95" title="Windows 95">Windows 95</a>, which tried to automate device detection and configuration as much as possible, but could still fall back to manual settings if necessary. During the initial install process of Windows 95, it would attempt to automatically detect all devices installed in the system. Since full auto-detection of everything was a new process without full industry support, the detection process constantly wrote to a progress tracking log file during the detection process. In the event that device probing would fail and the system would freeze, the end-user could reboot the computer, restart the detection process, and the installer would use the tracking log to skip past the point that caused the previous freeze.</p>
<p>At the time, there could be a mix of devices in a system, some capable of automatic configuration, and some still using fully manual settings using jumpers and DIP switches. The old world of DOS still lurked underneath Windows 95, and systems could be configured to load devices three different ways:</p>
<ul>
<li>through Windows 95 device manager drivers only</li>
<li>using DOS drivers loaded in the CONFIG.SYS and AUTOEXEC.BAT configuration files</li>
<li>using both DOS drivers and Windows 95 device manager drivers together</li>
</ul>
<p>Microsoft could not assert full control over all device settings, so configuration files could include a mix of driver entries inserted by the Windows 95 automatic configuration process, and could also include driver entries inserted or modified manually by the computer users themselves. The Windows 95 device manager also could offer users a choice of several semi-automatic configurations to try to free up resources for devices that still needed manual configuration.</p>
<div class="thumb tright">
<div class="thumbinner" style="width:222px;"><a href="/wiki/File:Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/b/ba/Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg/220px-Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg" width="220" height="360" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/b/ba/Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg/330px-Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/b/ba/Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg/440px-Kouwell_KW-524J_-_8-bit_PC_ISA_interface_card_-_4_port_-_2_serial_-_2_parallel.jpg 2x" data-file-width="1920" data-file-height="3144" /></a>

<p>Also, although some later ISA devices were capable of automatic configuration, it was common for PC ISA expansion cards to limit themselves to a very small number of choices for interrupt request lines. For example, a network interface might limit itself to only interrupts 3, 7, and 10, while a sound card might limit itself to interrupts 5, 7, and 12. This results in few configuration choices if some of those interrupts are already used by some other device.</p>
<p>The hardware of PC computers additionally limited device expansion options because interrupts could not be shared, and some multifunction expansion cards would use multiple interrupts for different card functions, such as a dual serial port card requiring a separate interrupt for each serial port.</p>
<p>Because of this complex operating environment, the <a href="/wiki/Autodetection" title="Autodetection">autodetection</a> process sometimes produced incorrect results, especially in systems with large numbers of expansion devices. This led to device conflicts within Windows 95, resulting in devices which were supposed to be fully self-configuring failing to work. The unreliability of the device installation process led to Plug and Play being sometimes referred to as <i>Plug and Pray</i>.</p>
<p>Up until about 2000, PC computers could still be purchased with a mix of ISA and PCI slots, so it was still possible that manual ISA device configuration might be necessary. But with successive releases of new operating systems like Windows 2000 and Windows XP, Microsoft had sufficient clout to say that drivers would not be provided for older devices that did not support auto-detection. In some cases, the user was forced to purchase new expansion devices or a whole new system to support the next operating system release.</p>
<h2><span class="mw-headline" id="Current_plug_and_play_interfaces">Current plug and play interfaces</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Plug_and_play&amp;action=edit&amp;section=9" title="Edit section: Current plug and play interfaces">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Several completely automated computer interfaces are currently used, each of which requires no device configuration by the computer user, and the only task of the user is to install software for the self-configuring devices.</p>
<ul>
<li><a href="/wiki/IEEE_1394" title="IEEE 1394">IEEE 1394</a> (FireWire)</li>
<li><a href="/wiki/Peripheral_Component_Interconnect" class="mw-redirect" title="Peripheral Component Interconnect">PCI</a>, <a href="/wiki/Mini_PCI" class="mw-redirect" title="Mini PCI">Mini PCI</a></li>
<li><a href="/wiki/PCI_Express" title="PCI Express">PCI Express</a>, <a href="/wiki/Mini_PCI_Express" class="mw-redirect" title="Mini PCI Express">Mini PCI Express</a></li>
<li><a href="/wiki/PCMCIA" class="mw-redirect" title="PCMCIA">PCMCIA</a>, <a href="/wiki/PC_Card" title="PC Card">PC Card</a>, <a href="/wiki/ExpressCard" title="ExpressCard">ExpressCard</a></li>
<li><a href="/wiki/USB" title="USB">USB</a></li>
</ul>
<p>For most of these interfaces, very little technical information is available to the end-user about the performance of the interface. Although both FireWire and USB have bandwidth that must be shared by all devices, most modern operating systems are unable to monitor and report the amount of bandwidth being used or available, or report activity on what devices are currently using the interface.</p>
